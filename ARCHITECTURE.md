# Architecture Notes

## Ingestion flow

The hot path is **non-blocking on the client and event-driven server-side**:

1. User submits a message → `Frontend` POSTs to `Chatbot API`'s SSE endpoint.
2. `Chatbot API` persists the user message, builds a windowed message history (last N turns), and calls `LLMClient.stream_chat(...)` from the SDK.
3. SDK wrapper opens a streaming connection to the provider (OpenAI / Anthropic / Mock). It yields token chunks back to the chatbot, which immediately forwards them to the client as SSE events.
4. As tokens flow, SDK accumulates: TTFT (first chunk timestamp - start), total latency, prompt/completion tokens (from provider usage chunk), input/output previews (truncated, PII-redacted).
5. **On stream end / error / cancellation**, SDK builds a `LogPayload` and `emit()`s it. `emit()` schedules an async POST to `Ingestion API /v1/logs` and returns immediately — the chatbot's `finally` block never blocks on the network.
6. `Ingestion API` validates the Pydantic schema and `XADD`s the payload to the `inference_logs` Redis Stream. Returns `202 Accepted` in <5 ms typical.
7. The separate `ingestion-consumer` worker (`xreadgroup`, consumer group `ingestion-workers`) drains the stream, runs a second PII pass, and `UPSERT`s into `inference_logs` (idempotent on `request_id`) plus an `INSERT INTO log_events` audit row. It `XACK`s only on successful commit.
8. `Frontend` Dashboard polls `Ingestion API /v1/stats/*` endpoints every 5s. Charts read directly from `inference_logs` aggregates.

Decoupling rule: **the chatbot never writes to `inference_logs`**. Only the consumer does. Likewise, the SDK never knows about Postgres.

## Logging strategy

| Concern | Decision |
|---|---|
| **Where it's captured** | In the SDK wrapper — single chokepoint, applies to every provider transparently. |
| **What's captured** | request_id, conversation_id, provider, model, status, latency_ms, ttft_ms, prompt/completion/total tokens, input_preview, output_preview, streamed, started_at, completed_at, error_type, error_message, JSONB metadata. |
| **When it's emitted** | In SDK's `finally` block — fires whether the call succeeded, errored, or was cancelled mid-stream. Status reflects which path. |
| **Sync vs async** | Async via `asyncio.create_task`. Chat never awaits the log write. Logger has a 2s HTTP timeout and swallows all exceptions (logs a warning to stderr). |
| **Idempotency** | `request_id` is generated by the SDK per call; ingestion does `INSERT … ON CONFLICT (request_id) DO UPDATE`. Safe to retry. |
| **Sampling** | None for v1 — every call is logged. Spec: would add `LOG_SAMPLE_RATE` env for high-volume settings. |
| **Levels / verbosity** | The full payload is always sent. Previews are bounded at 500 chars to prevent runaway storage on large prompts. |
| **PII** | Redacted in the SDK before send AND re-redacted in the consumer before write (defense in depth). |
| **Audit** | Every received payload also lands in `log_events` (append-only JSONB). Lets us replay or debug schema-drift incidents. |
| **Correlation** | `request_id` ties SDK call ↔ ingestion row ↔ consumer log line. `conversation_id` ties many calls together. |

## Scaling considerations

**What breaks first as load grows, and what we'd do:**

| Bottleneck (in order) | Symptom | Mitigation |
|---|---|---|
| 1. Single chatbot-api process | CPU saturated; SSE connections capped | Horizontal scale (multiple replicas behind a load balancer); move cancellation registry from in-memory to Redis pub/sub keyed by `conversation_id`. |
| 2. Synchronous DB writes from chatbot | Conversation/message inserts slow under bursty traffic | Already partially decoupled: only msgs hit chatbot's session, and inference_logs goes through the queue. Could push msg writes to the queue too. |
| 3. Ingestion API CPU | All it does is validate + XADD, so very high ceiling | Scale ingestion-api horizontally — stateless. |
| 4. Single consumer worker | Stream backlog grows | Scale consumer replicas — consumer group automatically partitions work. Each worker gets a distinct subset of pending messages. |
| 5. Postgres write throughput on inference_logs | Insert lag, vacuum pressure | Partition `inference_logs` by month on `started_at`; add a read replica for dashboard queries. |
| 6. Dashboard query latency at billions of rows | p95 query >1s | Tier hot/cold: keep last 7 days in Postgres, archive older to ClickHouse / S3+Athena. Pre-compute hourly rollups. |
| 7. Redis memory | Stream grows unbounded | Trim with `XADD ... MAXLEN ~ 100000` — once we've persisted to Postgres we don't need it in Redis anymore. |

## Failure handling assumptions

| Failure | What happens today | Why |
|---|---|---|
| **Ingestion API down** | Chat works normally. SDK logs a warning per failed POST; logs for those calls are lost. | Hard requirement: observability must never break product. v2 adds local disk buffer. |
| **Consumer worker down** | API still accepts logs (they queue in Redis). When worker comes back, it drains the backlog from where it left off (consumer-group offset). | At-least-once delivery; idempotent UPSERT covers the rest. |
| **Postgres down** | Consumer fails to commit, does NOT `XACK`, message gets re-delivered. Chatbot conversations can't be created/loaded (returns 5xx). | Right blast radius — DB outage is a real outage; don't pretend otherwise. |
| **Redis down** | Ingestion `/v1/logs` returns 503. SDK swallows, logs drop. Chat keeps working. | Same as ingestion-down case. |
| **Provider 5xx / timeout** | SDK catches the exception in `stream_chat`, yields an `error` event to the client, logs status=`error`, error_type, error_message. Chat UI shows an error bubble. | Provider failures are common; needs to be a clean code path, not a crash. |
| **User clicks Stop mid-stream** | `cancel_event.set()` is observed by the provider loop on the next iteration; SDK closes the upstream stream. Inference log written with `status='cancelled'` and the partial output captured. Conversation status flipped to `cancelled` (next user message flips it back to `active`). | Cancellation must be observable in analytics. |
| **Duplicate log** | `INSERT … ON CONFLICT (request_id) DO UPDATE` — second insert updates the existing row with the latest snapshot. No duplicate analytics. | Idempotent by design. |
| **Long conversation exceeds context window** | SDK sends only the last `CONTEXT_WINDOW_TURNS` (default 10) message pairs to the provider. Older turns stay in DB and are visible in the UI. | Predictable token cost. Spec'd improvement: summarize older turns into a synthetic system message. |
| **No OPENAI_API_KEY** | SDK auto-falls back to `MockProvider`. Chat works end-to-end, logs flow, dashboards populate. | Reviewer can run with zero secrets. |

## Service boundaries (who owns what)

- **SDK** owns: capturing call metadata, PII redaction (first pass), shipping logs.
- **Chatbot API** owns: conversation/message persistence, context-window construction, SSE streaming, cancellation registry.
- **Ingestion API** owns: payload validation, queue ingestion (no DB writes from here).
- **Consumer** owns: queue draining, PII redaction (defense-in-depth pass), DB writes (inference_logs + log_events).
- **Frontend** owns: rendering, user input, calling the two APIs (chatbot for chat ops, ingestion for stats).

This separation is enforced by network boundaries (separate containers, separate ports) so coupling violations would be visible immediately.
